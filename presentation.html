Crowd Simulation – Budapest

Architecture & MCP-Driven Analytics

Use ← → or buttons to navigate

What the System Does

  • Simulates crowds moving toward event venues on a Budapest map
  • Groups of people walk a road graph, wander, then rally to venues
  • Cell towers (venues) have fixed capacity & dynamic occupancy
  • A min-cost flow algorithm assigns every person to the best tower each tick
  • Assignment data is served via HTTP API and MCP tools

System Architecture

Data Flow

Simulation Server

  • Runs the tick loop & road-graph movement
  • Computes min-cost flow assignment each tick
  • Exposes /api/groups with groups, venues, assignments

MCP Servers

  • Fetch live snapshot from the simulation API
  • Serve statistical data about people's locations
  • Provide tower assignment breakdowns
  • LLM calls tools with zero input — data is always live

MCP Tools – Statistical Data

get_assignment_group_level

  • One entry per (group, tower) pair
  • Fields: groupIndex, venueId, count
  • Shows how many people from each group are assigned to each tower
[
  { "groupIndex": 0, "venueId": 1, "count": 12 },
  { "groupIndex": 1, "venueId": 2, "count": 8 }
]

get_assignment_person_level

  • One entry per individual person
  • Fields: groupIndex, venueId
  • Expanded view — each count becomes N identical rows
[
  { "groupIndex": 0, "venueId": 1 },
  { "groupIndex": 0, "venueId": 1 },
  ...  // 12 entries for group 0 → venue 1
]

LLM Integration via MCP

The LLM never touches the simulation directly — it uses MCP tools to get statistical data about people's locations and tower assignments, then reasons over it.

Key Takeaways

  • Simulation server owns the state: movement, road graph, min-cost flow
  • MCP servers are stateless proxies that provide statistical data on demand
  • Two granularities: group-level aggregates and person-level detail
  • LLM consumes live location & assignment stats through MCP protocol
  • Clean separation: simulation ↔ data access ↔ AI reasoning
