CDN Orchestration via MCP

Using crowd-location intelligence to deploy CDN servers optimally

← → or buttons to navigate

The Problem

  • Subscribers (people) are constantly moving across the city
  • CDN servers must be placed close to where demand actually is
  • Static placement wastes capacity in empty areas and starves crowded ones
  • We need real-time location data to drive dynamic CDN deployment

The Solution

  • MCP Server exposes live subscriber location & tower-assignment statistics
  • An Orchestration Agent (LLM) consumes this data via MCP tools
  • The agent analyses crowd density and decides optimal CDN placement
  • CDN servers are deployed nearest to the highest subscriber concentrations

Architecture

How the Agent Decides

  • 1Calls get_assignment_group_level — gets crowd-to-tower counts
  • 2Calls get_assignment_person_level — gets per-subscriber locations
  • 3Identifies density hotspots — which towers have the most subscribers
  • 4Computes optimal CDN positions — nearest to the highest concentrations
  • 5Deploys CDN servers at those locations, tears down underutilised ones
  • 6Repeats on a schedule — CDN topology follows the crowd in real time

MCP Data → CDN Decisions

What MCP Provides

  • Which tower each subscriber is assigned to
  • How many subscribers per tower (load)
  • Group positions → geographic demand map
  • All data is live — refreshed every tick

What the Agent Does With It

  • Ranks towers by subscriber count
  • Maps tower coordinates to candidate CDN sites
  • Balances capacity: avoids over-provisioning quiet areas
  • Shifts CDN resources as crowds move between venues

Continuous Orchestration Loop

The loop runs continuously — as subscribers move, the MCP server reflects new density data, and the orchestration agent repositions CDN servers to stay optimal.

Key Takeaways

  • MCP Server bridges live crowd simulation data to the AI agent
  • The orchestration agent uses subscriber location stats to find density hotspots
  • CDN servers are deployed nearest to where subscribers actually are
  • As crowds shift, the agent continuously re-optimises CDN placement
  • Result: lower latency, better capacity utilisation, dynamic infrastructure
